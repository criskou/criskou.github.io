性能测试工具JMeter

Jmeter工作原理

基本使用

  1. 添加线程组；
  2. 添加取样器-Http请求，填写协议、服务器IP、端口号（HTTP8080）、请求方式、url、编码方式（默认为ISO国际标准，根据需要设置为utf-8）、请求参数；
  3. 在线程组下添加查看结果树；
  4. 启动测试，查看结果。

重点组件

1、线程组

线程数：可以理解为用户数；

Ramp-UP时间：设置测试运行时间；

循环次数：脚本循环执行次数，如果设置为永远，需要结合调度器使用（持续时间：脚本执行时间；启动延迟：等待若干秒后开始执行）。

2、Http取样器

3、查看结果树

4、HTTP Cookie管理器

添加Cookie管理器后会自动管理Cookie。

每个线程都有自己的“Cookie存储区”，如果要查看，Cookie管理器里是看不到的，可以使用查看结果树查看。

5、HTTP请求默认值

把HTTP取样器中相同的部分抽离出来放到HTTP请求默认值元件中，在其他请求中就可以省略不写这部分内容了，提高效率。

添加方式：线程组-添加-配置元件-HTTP请求默认值。

6、用户定义变量

添加方式：同样是在配置元件里。

引用方式：${变量名}。

7、CSV数据文件设置

设置完之后，就可以使用${变量名}引用数据文件。

8、JSON提取器、正则表达式提取器

添加方式：线程组-后置处理器-JSON提取器

JSONPATH参考链接

在查看结果树中切换为JSON Path Tester可以查看所取结果是否正确。

注意元件顺序，JSON提取器的提取顺序是从上到下的，所以举个例子，产生token的登录请求应该排在其他需要token的请求之前。

9、JSON断言

Assert JSON Path exists：填写JSON Path，即需要断言的内容；

Additionally assert value：如果不勾选，就可以用来断言字段是否存在，如果勾选则需要填写Expected Value；

Match as regular expression：不勾选时预期值必须写完整，勾选则是模糊匹配，仅写上部分关键词就可以断言成功。

Invert assertion：结果翻转，如果断言成功，则返回失败，反之相反。

10、同步定时器

当多个线程同时启动时，它们可能会在不同的时间间隔内执行，同步定时器的作用是设置一个集合点，当线程数达到指定值后，再一起释放，这样就保证能在同一个时间点发送请求，但这样还是不能保证请求同一时刻到达服务器，所以只能说是较真实地模拟用户并发访问的场景。

同步定时器设置模拟用户数量最好是和线程组里的线程数量相同，或者线程数量是模拟用户数量的整数倍（但线程循环要设为永远，指定时间循环），这样才不会导致有剩余的请求发送不出去（因为要等到达集合点的数量，才能把这些请求一起发送出去）。

11、事务控制器

事务控制器可以将多个接口划归为一个事务，相当于模拟用户一系列动作。

添加之后在聚合报告中就能直观地看到这个事务的测试结果了。

常用插件

按照链接下载Plugings Manager，放到lib-ext文件夹下。

Stepping Thread Group（梯度线程组）

每轮跑完后，增加若干线程继续测试。

主要参数设置：

  • Next，add（）threads every（）seconds：每多少秒增加多少个线程；

  • using ramp-up[^1]：启动线程的时间；

  • then hold load for：线程全部启动完保持多长时间；

  • finally stop （）threads every （）seconds：每几秒停止几个线程。

监听器

Active Threads Over Time

实时显示运行的线程数

Response Times Over Time

响应时间随时间变化图，可以表示不同时间点的响应性能，从而发现可能存在的性能问题和瓶颈。

Transactions per Seconds（TPS）

TPS表示每秒处理事务数，这个指标反映了系统在同一时间内处理业务的最大能力，TPS越高，说明系统处理能力越强。

测试报告

性能分析

响应时间

如果响应时间超标，说明系统到达了瓶颈。

错误率

高并发场景下，系统能否正常处理业务。（要求99.99%-99.999%）

错误率高的原因：

1、接口请求错误。

2、服务器无法继续处理，达到了瓶颈（代码写的不好，内存泄露，硬件资源等）。

3、后端系统限流（系统里配置了不能超过多少并发）、熔断、降级。

（熔断：防止系统因某个服务器的故障而整体崩溃。当电商平台上用户支付时，收银台发现某个支付渠道，如微信支付失败率突增，超时严重，那么就可以临时把这个支付方式熔断掉。

降级：主动关闭一些非核心功能，以确保核心功能的正常运行。某次腾讯视频挂了的时候，用户名称默认显示未腾讯用户，这也是一种降级方式，用兜底名称做展示。）

吞吐量

吞吐量越大，性能越好；吞吐量相对稳定或者变低，可能系统达到了性能瓶颈。

吞吐量变化规律：

波动很大：代表系统性能不稳定。

慢慢变高，在趋于稳定：和并发量相关。如果并发量小于吞吐量，慢慢增大并发量，吞吐量也会随之增加。

慢慢变低，并发量也减少了：要么性能测试要结束了，并发减少；也可能是系统变的卡顿，从而导致响应时间变慢，导致单个线程发起的并发量变少。

[^1]:rump up 逐步增加
